開発の目的

テーマや条件を入力すると、幼児が読むような簡単な物語を生成するwebアプリケーションを作ります。全世界の子供を持っている親に向けてリリースします。

この開発の目的は、開発の一連の作業（工程）をすることにより、かかる開発要員が品質の高いソフトウェアを念頭に置き、デザインとプログラミングのスキルを向上させることにある。

また、モダンなフレームワークを使うことで新たな技術をキャッチアップする。フロントフレームワークをReactとし、バックエンドとデータベースは、FireBaseに任せるものとする。インフラはGCPを用いる。

今回用いるReactはコンポーネント志向である。コンポーネント志向とは、アプリケーションの構造やデザインを独立した再利用可能な部品（コンポーネント）に分割するソフトウェア開発のアプローチです。このアプローチは、アプリケーションのコードの管理、維持、再利用を容易にし、開発チームの効率を向上させることを目的としています。

コンポーネント志向開発の主な特徴は以下の通りです：

再利用可能性： コンポーネントは独立した機能を持ち、アプリケーション内の他の部分と疎結合になっているため、他の箇所で再利用しやすいです。

モジュール性： アプリケーションは小さな単位のコンポーネントに分割され、各コンポーネントは特定の機能や目的に焦点を当てています。これにより、アプリケーション全体の構造がわかりやすくなり、コードの品質や保守性が向上します。

分離の原則： コンポーネント志向開発では、関心事を分離することが重要です。つまり、各コンポーネントは自分自身の機能に集中し、他のコンポーネントとは最小限の接触を持つことが求められます。

データフローの一方向性： コンポーネント間のデータフローは一方向になることが推奨されます。これにより、データの流れが予測可能であり、アプリケーションの状態管理が容易になります。

Reactは、コンポーネント志向開発をサポートするフロントエンドライブラリの一つです。Reactでは、UIをコンポーネントという再利用可能な要素に分割し、それらを組み合わせることでアプリケーション全体を構築します。これにより、開発者はコードの再利用性や可読性を向上させ、効率的な開発が可能になります。

使用する技術

React ユーザーインターフェイスを構築するためのJavaScriptライブラリ。コンポーネントベースのアーキテクチャで開発を行います。
TypeScript 型安全性を提供するJavaScriptのスーパーセット。開発効率やコード品質の向上が期待できます。
Firebase Authentication ユーザー認証機能を提供し、サインアップ、ログイン、パスワードリセットなどの機能を実装します。
Firestore ユーザーのプロフィール、生成された物語データ、カスタマイズ情報などを保存します。リアルタイム同期が可能で、データの変更を即座に反映できます。
Firebase Storage 画像や音声ファイルなどのメディアデータを保存するために使用します。これにより、絵本のカスタマイズや読み聞かせ機能を実現できます。
Firebase Hosting 静的ファイル（HTML, CSS, JavaScript, 画像ファイルなど）をホスティングするサービス。グローバルCDNと無料のSSL証明書が提供されています。
Custom Domain (オプション) 独自ドメインでアプリケーションを公開することができます。
Firebase CLI デプロイや開発を効率化するためのコマンドラインツール。ローカル開発環境から簡単にデプロイが行えます。
OpenAI API GPT-3.5-turboモデルを利用して物語を生成します。フロントエンドからリクエストを受け取り、生成した物語を返すようにします。

これらの技術を組み合わせることで、効率的かつスケーラブルなシステム設計が可能です。また、サーバーレスアーキテクチャを使用することで、コスト効率の高いインフラ運用が実現できます。

機能要件

(Must)ユーザー認証 サインアップ、ログイン、パスワードリセット機能
(Must)物語生成 ユーザーが入力したテーマや条件に基づいて、AIが物語を生成する機能
(できればやってみたい)PDF生成 生成した物語をPDFとして出力する機能
物語のシェア機能 SNSやメールで生成された物語を共有できる機能
読み聞かせ機能 生成された物語を音声で読み上げる機能
絵本のカスタマイズ機能 イラストやフォントなどを選択して、物語の見た目をカスタマイズできる機能
マルチ言語対応 アプリ内で複数の言語を選択できる機能
プラン管理 無料プランと有料プランを管理し、アップグレードを促す機能
ユーザープロフィール管理 ユーザー情報の編集や閲覧ができる機能

非機能要件

レスポンシブデザイン PC、タブレット、スマートフォンなど、さまざまなデバイスで適切な表示がされること
パフォーマンス 高速な物語生成、読み込み、カスタマイズ機能が実現されていること
セキュリティ ユーザーデータが安全に保管され、不正アクセスやデータ漏洩がないこと
スケーラビリティ 大量のユーザーやリクエストにも対応できるシステムが構築されていること
メンテナンス性 開発チームが容易に機能追加や修正ができるアーキテクチャが構築されていること
ユーザビリティ 直感的で使いやすいUIが実現されていること
ローカライゼーション 地域や言語に応じて適切なコンテンツが提供されること
クロスプラットフォーム対応 iOS、Android、Webブラウザなどで利用できること
アクセシビリティ 視覚・聴覚・操作上の障害を持つユーザーにも利用しやすい設計がされていること

基本方針

開発に関わるドキュメントおよびプログラムのすべてをHTML文書にしてWebブラウザからアクセス可能にすること。また、アプリケーションのインストールや取り扱いを説明したマニュアルもHTML文書として作成すること。すなわち、この文書からたどれる全文書（このページの最上部にリンクされた全ページ）を完成させなければならない。以下にそれぞれの基本方針を記載する。

  • 設計方針

    アプリケーションは一つのページで構成され、各機能はそのページ内で実現する。さまざまなデバイス（PC、タブレット、スマートフォン）で適切な表示がされるよう、レスポンシブデザインを実装する。高速なページ読み込みや物語生成を実現するために、パフォーマンス最適化を行います。キャッシュやコードの最適化、遅延読み込みなどの手法を適用する。

  • 管理方針

    開発に伴う仕様書・設計書・報告書などの全ドキュメントおよび全ソースプログラムをGitの中に格納し、リビジョン管理（バージョン管理）を行い、開発チームで共有すること。

  • 実装方針

    コーディング規約を遵守すること。細かくはeslint, prettierに記述する。コンポーネント名はアッパーキャメルケース、変数名・関数名はローワーキャメルケース、定数名はスネークケースを用いること。

  • テスト方針

    ユーザの操作に関わるテストケースは設計段階でテスト仕様書にまとめること。Jestによるテストケースを行うこと。テスト結果は報告書にまとめ、各々のリリースや大きな修正の直後に全件実施（回帰テスト）を行わなければならない。

  • リリース方針

    （Major.Minor.Patch）を使用して、リリースの互換性や変更の大きさを明示します。Gitフローに従って、リリースブランチを作成し、リリース用のバグ修正やドキュメントの更新を行います。リリースごとに、変更履歴（CHANGELOG）を更新して、新機能や修正、削除された機能などの詳細を明記します。自動化されたユニットテスト、統合テスト、エンドツーエンドテストを実施し、リリースの品質を確保します。また、リリース前にコードレビューや機能検証を行います。リリースに伴うドキュメントの更新が重要です。プロジェクトに適切なライセンスを適用し、著作権表示を含めます。

基本要件

以下に開発計画書および基本設計書を作成するにあって必要なことを記載する。

  • アプリケーションの用途

    寝る前の子供に読み聞かせするための絵本を生成するアプリケーションを作成する。

  • 対象となるユーザ

    現在、世界には約21億の子供がいるとされています。このアプリケーションは、幼児向けの物語を生成することができるため、対象年齢は2歳から6歳までと考えられます。この年齢層は、約6億人の子供たちが対象となります。また、親である約12億人も潜在的な顧客となります。

  • ハードウェア構成

    Name: Apple Macintosh シリーズ　(MacBook / MacBook Air / MacBook Pro / Mac mini / iMac / Mac Pro など)
    Processor: Intel Core 2 Duo 2GHz クラス以上（64bit アーキテクチャの Xenon などを含む）
    Memory: 1GB 以上
    Hard Disk: 250MB 以上の空き容量

  • ソフトウェア構成

    OS: OS X Yosemite (10.10) / OS X El Capitan (10.11) / macOS Sierra (10.12) 以上
    開発に用いるプログラミング言語・フレームワークは以下とする。
    1) React 18*
    2) TypeScript 4*

  • 目標と性能

    ユーザがログインし、条件指定することで物語を生成を可能とする。また、物語の履歴確認機能やPDF生成も可能とする。

  • 制限事項

    標準ライブラリ（必要であれば拡張ライブラリ）を使用（援用）して開発すること。Macintoshシリーズをターゲットにしているが、実装に用いる言語はプラットフォーム非依存性を仮想マシンによって実現しているプログラミング言語である。したがって、開発したアプリケーションはLinuxやWindowsなどでも動作することが望ましい。

開発体制

開発に関わるメンバの一覧を作成し、プロジェクトマネージャ、デザイナ、プログラマ、テスタなどの役割を明確化し、加えて、開発するコンポーネント毎に担当者を決めて、それらを開発計画書の中に明記すること。また、仕様適合検査の責任者を立て、次項の開発スケジュールに示されたマイルストーン（目標達成へ向かう過程の一つの段階）ごとに、設計段階から仕様のチェックをクライアント（開発依頼者）と共に行うこと。開発にあたって不明なことが生じた場合には、必ずクライアントと協議して事を進めなければならない。

開発スケジュール

開発スケジュールを前項の開発体制と共に開発計画書の中に明記し、着手する前にクライアントの認可を受ける（契約を取り交わす）こと。実際の開発は1週間で行うこと。1週間以内での開発（納品＆リリース）を厳守しなければならない。
